# Factor V: Safe Retries

*Every mutation is safely retried by a probabilistic caller*

A deterministic service client retries with the same request bytes, following a fixed backoff schedule, and stops after three attempts. A probabilistic caller retries the same *intent* through a fresh generation — producing a structurally different but semantically equivalent request. It may retry because it lost context and re-initiated an operation it already completed. It may retry through a different tool entirely, calling `create_order` after `reorder_last` timed out. In a multi-agent system, two agents may independently conclude the same operation is needed and both initiate it. The retry surface is not bounded by code. It is bounded by nothing. The caller is optimized for helpfulness over accuracy; its default response to ambiguity is to try harder, not to stop.

Traditional idempotency patterns — Stripe's idempotency keys, AWS SQS deduplication tokens, the IETF Idempotency-Key header — assume the caller generates one token per logical operation, persists it across retries, and sends identical parameters each time. LLMs satisfy none of these assumptions. They have no durable state between invocations. They cannot distinguish a retry from a new request. And when an idempotency token appears as a tool parameter, they fabricate one — generating a syntactically valid UUID with complete conviction that binds to no prior operation and guarantees nothing.

**The control plane owns deduplication and compensation because the reasoning layer cannot — it has no durable state, no deterministic output, and no concept of "this is a retry."** One proven pattern: the control plane ensures a deduplication key is derived deterministically — whether computed server-side from the operation's semantics or generated by a deterministic step the caller must invoke before the mutation — rather than trusting the reasoning layer to produce one. Stripe deduplicates the charge. Salesforce deduplicates the contact creation. ServiceNow deduplicates the case. NetSuite deduplicates the journal entry. The key is derived at the boundary between reasoning and execution, where the operation's identity can be determined before it reaches any backend system.

A checkout that succeeds in Stripe but times out before the PMS room status update must not charge the guest twice when the caller retries — and the caller will retry, because it cannot distinguish a timeout from a failure. The control plane sequences each step with its own deduplication key and defines compensation — reverse the Stripe charge, release the inventory hold in the warehouse management system, void the folio in the PMS — before the first step executes. When the trigger for compensation is a probabilistic caller that doesn't know it's retrying, the compensation path cannot be an afterthought discovered during incident response. It is defined at design time and tested like any other code path.

When the control plane has no deduplication layer, every retry is a new operation. Agents fabricate fresh idempotency tokens; backends process duplicate charges. A three-step workflow fails on step two and retries from the top, creating orphaned records that surface as double billing notifications a week later. The agent reports success each time. The architecture has no way to detect that the same intent produced two consequences. When the caller is probabilistic, idempotency is not a resilience optimization. It is the control plane's guarantee that the caller cannot corrupt state by doing what probabilistic callers do — retrying.
