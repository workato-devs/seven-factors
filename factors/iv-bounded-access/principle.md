# Factor IV: Bounded Access

*Each caller sees only the capabilities its role requires*

Most teams building agentic applications expose the full namespace to the agent. The agent gets everything it might need — and the system prompt or the agent's own judgment determines what it should and shouldn't use. This is not access control. It is the absence of access control, masked by the agent's willingness to follow instructions.

When teams do recognize tool access as a security surface, the most common response is prompt-based restriction — and it is worse than no strategy at all, because it creates the illusion of control. A system prompt that says "only take actions the user has authority to authorize" or "verify identity before acting on sensitive requests" places authorization decisions inside the same token stream that processes user input. Prompt injection attacks bypass these restrictions at rates between 25% and 92% in controlled benchmarks. The TIP exploitation paper achieved remote code execution on every tested agent-LLM pair through crafted tool descriptions and return values. The model that follows the access control instruction is the same model that follows the injected instruction. This is not a deficiency to be solved by better models. It is the confused deputy problem applied to a system that cannot distinguish instructions from data.

**The blast radius of a compromised agent is exactly the tools in its namespace.** The architectural response is to make that namespace as small as the caller's role requires. The strength of the boundary depends on where enforcement lives. A tool list hardcoded in application code — the first step past the wide-open default — is correct only as long as it matches the authorization requirements of every caller who reaches that agent — and nothing validates that it does. Gateway-level filtering — Kong Tool ACLs, Cloudflare's WorkOS integration — reduces what the agent can discover, but the tools still exist on the upstream server. Structural separation, where different callers connect to entirely different server instances, provides the strongest guarantee: capabilities that do not exist in the caller's runtime cannot be reached regardless of policy state. A guest-facing agent connects to an MCP server exposing `search_bookings` and `submit_service_request` — tools scoped to the authenticated guest's own records in Salesforce and the PMS. A staff-facing agent connects to a separate server exposing `override_rate`, `comp_room`, and `view_all_guests` — tools carrying staff-level authorization that can operate across any guest's records, process rate adjustments through Stripe, and write to audit-sensitive CRM fields. Even operations that sound similar — searching bookings — are different capabilities encoding different authorization boundaries. The guest agent cannot escalate to staff operations because those operations do not exist in its namespace.

The same structural principle applies when agents delegate to other agents — A2A's Extended Agent Cards advertise different capabilities to different authenticated callers, bounding the blast radius at the skill boundary. For high-consequence systems — PCI-scoped payment processing, HIPAA-regulated clinical operations — compound enforcement adds declarative policy evaluation through Cedar or Cerbos within each structurally separated namespace. Factor I governs what happens when a tool is invoked. Factor IV determines whether the tool exists in the caller's world at all.

Without an architectural access boundary, every agent in the system sees every tool. Implementation-leaked tool names — the failure mode Factor III identifies — advertise the backend architecture; credentials embedded in tool configurations hand attackers keys alongside the map. In this way, prompt injection is the escalation vector allowing a compromised customer-facing chatbot the same tool access as an internal operations agent. A litmus test for whether access boundaries are truly architectural: can agentic system prompts leak — and tool lists be published — without compromising access control? Least privilege is not a policy an agent follows. It is a property of the control plane itself.
