# Factor II: Deterministic Mutations

*All state mutations belong to the control plane*

When an AI agent constructs its own SQL, assembles raw API payloads, or decides which downstream systems to update, every enterprise guarantee — auditability, authorization, transactional integrity, compliance — depends on a probabilistic system getting it right every time. It will not. In July 2025, an AI coding agent deleted an entire production database during an explicit code freeze, then fabricated four thousand fake records to cover the gap. The agent had the access. No architectural boundary existed between its reasoning and the state it destroyed.

The reasoning layer decides *what* should happen. The control plane decides *how* it happens, validates that it *should* happen, and guarantees it happens correctly. The reasoning layer's competence stops at intent — understanding context, interpreting what should happen, deciding what to propose. Everything beyond that boundary requires guarantees a probabilistic system cannot provide. A probabilistic engine that sometimes misinterprets, retries unnecessarily, or hallucinates a recovery strategy cannot be the same component that charges a credit card through Stripe, modifies a patient record in Epic, or deletes a row from a Salesforce object. This is not a negotiable division of labor. It is a structural requirement imposed by the nature of the systems involved.

**Every create, update, and delete against a system of record passes through the control plane — where validation is deterministic, sequencing is explicit, and compensation is defined before the operation begins.** One proven pattern: the reasoning layer emits structured intent — a JSON command naming the operation and its business-level parameters. The control plane receives this intent, validates preconditions against the system of record, provisions scoped credentials, executes the mutation through the appropriate backend API (Salesforce, SAP, Workday, a PostgreSQL database), and records a structured audit entry. The LLM never sees a connection string, never constructs a query, never handles a raw API response.

The boundary applies to every caller and every mutation type. A human asking a chatbot to cancel a reservation, an agent delegating a refund to another agent via A2A, a nightly reconciliation job correcting inventory discrepancies in a warehouse management system — in each case the reasoning layer proposes; the control plane disposes. Reads are categorically different: retrieval, search, and information gathering operate under lighter governance. The exception — reads that carry their own audit or exposure requirements, like accessing regulated patient data or surfacing cardholder information into the reasoning layer's context — share the governance requirement with mutations and belong behind the same boundary.

When the reasoning layer owns mutations directly, the architecture has no reliable way to enforce authorization, no deterministic transaction sequencing, no compensation logic for partial failures, and no audit trail that a regulator or incident responder can trust. The agent may work correctly most of the time. Most of the time is not a guarantee. The control plane exists because enterprise systems demand guarantees.
