# Factor VI: Recovery Contracts

*The reasoning layer never guesses at state*

When an operation fails and returns a raw HTTP 500, a stack trace, or a string that says "something went wrong," the reasoning layer does not stop. It fills the information vacuum with imagination — confabulating a recovery strategy, retrying with modified parameters, or inventing a result. This is not a deficiency in any particular model. It is the predictable behavior of a system optimized for helpfulness when the control plane gives it nothing to act on.

A recovery contract is a structured response that tells the caller what happened, whether recovery is needed, and what to do next. A checkout that fails because of an outstanding balance is not the same as one that fails because the payment processor is unreachable, which is not the same as one that fails because the reasoning layer sent malformed parameters. Each demands a different caller response. An HTTP 500 communicates none of these distinctions.

**When recovery is architectural, every capability invoked returns a structured result that separates outcome from description, declares the viability of operational recovery, and provides the caller's next action.** One proven pattern: a discriminated result envelope carrying a success/failure flag, a machine-readable error code (OUTSTANDING_BALANCE, PAYMENT_PROCESSOR_UNAVAILABLE, INVALID_PARAMETERS), a recoverability signal (retry, escalate, fail), and a suggested next action in prose. MCP's `isError` flag is the first step; AWS Bedrock's REPROMPT/FAILURE distinction is a second; Cloudflare's RFC 9457 extension — with `retryable`, `retry_after`, `owner_action_required`, and `error_category` fields — is the most complete production implementation to date. The specific schema is replaceable. The architectural commitment is not: the control plane tells the caller what to do, not just what went wrong.

Every error response is an open channel from a backend system into the reasoning layer. An unstructured channel carries two risks, not one. The reliability risk: agents that receive ambiguous errors confabulate recovery strategies. Structured feedback reduces that confabulation by 26–30% in controlled experiments. The security risk: free-text error messages are an active injection vector. A `retryable: boolean` field cannot carry a prompt injection payload. Research on tool invocation prompt exploitation has demonstrated that crafted tool responses — including error messages — can hijack agent behavior across major frameworks. Typed, validated fields constrain what flows from tool to agent, reducing the attack surface for every invocation.

When recovery contracts are absent, a delegating agent receives a failed A2A task with `{state: FAILED, message: "Something went wrong"}` — fully protocol-compliant, entirely useless — and must decide whether to retry, escalate, or abandon using nothing but inference. Ambiguous errors trigger retry storms that compound into the idempotency failures Factor V addresses: the duplicate charges, the orphaned records, the double billing that surfaces a week later. The recovery contract is not a convenience for the reasoning layer. It is the control plane's last opportunity to keep a probabilistic caller on a deterministic path — and off a costly one.
