# Factor III: Intent-Based Communication

*Tool boundaries follow intent, not implementation*

The agentic protocol stack — MCP for tool integration, A2A for agent-to-agent communication — defines how tools are discovered and how agents exchange tasks. Neither protocol constrains what happens at the boundary between a probabilistic caller and the deterministic systems it operates on. MCP will transport a tool named `stripe_create_charge` as readily as one named `check_out_guest`. A2A will delegate a task to a skill that requires a Salesforce Contact ID as willingly as one that accepts an email address. The protocol is not the problem. The interface the developer builds on top of it is.

An implementation-leaked **design** asks the reasoning layer to know things it should not know — which vendor processes payments, which system holds guest records, which API call produces a folio. It turns tool selection into a system integration exercise. The caller must map vendor-specific endpoints to outcomes, chain operations in the correct order, and pass system identifiers between steps. This is work that deterministic systems do reliably and that probabilistic systems do poorly — not because LLMs lack capability, but because orchestrating raw backend operations is not reasoning. It is plumbing.

**An intent interface names what the caller is trying to accomplish, accepts only identifiers the caller should possess, and hides everything the architecture needs to protect — not just for security, but because a probabilistic caller will reason about every detail it can see.** `check_out_guest` accepts a booking reference and a payment confirmation flag. `order_medication` accepts a patient identifier, medication name, and dosage. `reroute_shipment` accepts an order reference and a new destination. Each is a single operation that expresses one business outcome. The caller doesn't know — and cannot discover — whether checkout involves Stripe, whether medication ordering touches Epic, or whether rerouting calls FedEx and then UPS. An intent interface accepts **contextual identifiers** — an email address from a guest interaction, a booking reference from a task delegation, a service name from an alert — not vendor-specific primary keys. It expresses a domain-level outcome — not a system-level operation. It hides enough that a backend substitution changes nothing the caller sees. **The control plane translates in both directions: contextual identifiers in from the caller, domain-level outcomes back — confirmation that the guest is checked out, not a Stripe charge ID and a PMS status code.**

Behind the intent boundary, the control plane decomposes these interfaces into whatever vendor-specific **operations** the backend requires. `check_out_guest` may call the Stripe Charges API, update a Salesforce Contact record, change the PMS room status, generate a folio, and send a confirmation email. These **operations** — Stripe API calls, Salesforce SOQL queries, PMS status updates, EHR record modifications — are implementation details that belong behind the boundary. They are visible to the control plane, testable in isolation, and replaceable without touching the caller's interface. What the reasoning layer sees is a single tool named for a business outcome, accepting domain-level identifiers, hiding everything else. A well-designed tool is indistinguishable from what A2A calls a skill.

When a tool list reads like a list of exposed endpoints — `stripe_create_charge`, `salesforce_update_contact`, `pms_set_room_status` — the reasoning layer has been asked to do work that deterministic systems do reliably and that probabilistic systems do poorly. When it is intent-aligned — `check_out_guest`, `order_medication`, `reroute_shipment` — the **control plane** absorbs it. Every tool boundary is a communication architecture between a probabilistic caller and deterministic systems. The question is which side absorbs the complexity.
